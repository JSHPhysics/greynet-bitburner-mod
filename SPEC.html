01What it is

Greynet (working title) is a single-file HTML prototype for a hacking game with a clear conceit: a contract killer who works by writing code. The target is a person with a cybernetic heart implant. The brief is to make the heart fail in a way that reads as natural decline, so the autopsy doesn't flag it as murder.

Everything happens in one window: an ASCII drawing of a heart on the left, a terminal at the bottom, a telemetry sidebar on the right, and a stack of modal dialogues for the puzzles, editors and end-of-run screens.

The whole prototype is one HTML file — index.html, 2,391 lines — with no build step, no dependencies, and a small JSON blob in localStorage for save state.

Genre note. The file's name is greynet-bitburner-mod. The intended home is the JavaScript-driven hacking game Bitburner, whose own scripting API is called ns — that's why this prototype's automation API is called ns.greynet. For the prototype itself, all of this runs standalone in a browser.

02The player loop

Each contract follows the same shape. The intro modal walks the player through it; the terminal commands that drive it are scan, breach, cd, cat, nano, inject, run.

  1. Scan — list the four nodes (sub-controllers) on the target's implant. All start SEALED.
  2. Breach — for each node, the port has leaked some input/output pairs from its handshake. Write a one-line function unlock(n) that reproduces the hidden rule. On success the node becomes BREACHED.
  3. Read the docs — once breached, a node exposes a few files. The technical reference files explain how that subsystem works, what its healthy range is, and what trips the trace meter.
  4. Defeat countermeasures — two nodes hide a config file that keeps the heart healthy. Edit it with nano to disarm.
  5. Write the library — inject opens an editor for regulator.js. Define regulate(t); the routine calls it once per cardiac cycle.
  6. Run — drive the heart into arrest while keeping the cardiac log looking like a natural decline. Five clauses must all hold at the moment the heart stops.
  7. Earn the script — beat the contract by hand once, and the automation toolkit unlocks. The script command opens solver.js; from there, every manual action becomes a method on ns.greynet.

Each completed run is scored on four axes — time, trace, RAM, lines of code — and the personal bests are persisted so the player has something to beat next time.

04State model

The prototype keeps no class structure — it's all plain objects and free variables on a single script scope. There are two kinds of state: progress (the stuff that persists between sessions) and physiology (the live simulation, which resets every run).

The two halves matter:

VariableShapeWhat it is
hw {bpm, rhythmVar, amp, cond} The hardware readings. What the telemetry sidebar displays. What the player's regulate() sees as input.
setp {rate, gain} The setpoints the routine is currently asking for. hw drifts toward these each tick.
stream [{bpm, rhythmVar, amp, cond}, ...] The full per-cycle history. The signature check reads it; the real-time cliff check reads its last two entries.
reviewVal 0–100 The trace meter. Rises when requested rate jumps; falls slowly otherwise.
tick integer Cycles elapsed this run. Budget is 2,400.
beatPhase float Animation clock. (beatPhase % 1) is "where in this beat are we"; the integer part counts beats since the run started, and the bloodflow trace reads it directly.
playerRegulate function or null The compiled regulate(t) the routine calls each tick. Created by new Function(libraryText) on Run.
One important split. Setpoint vs reading. The script asks for a rate (setp.rate); the body delivers a bpm (hw.bpm) that drifts toward it. The trace meter watches the gap between the two — that's the whole mechanic.

05The four nodes

Each of the four sub-controllers is a node with the same shape: a breach puzzle, a set of files, optionally a countermeasure config that needs disarming. Differences are in what telemetry field the node controls and what countermeasure it ships with.

IDNameControlsBreach ruleFiles
n_rate RATE CTRL BPM. Reads setp.rate. n * 2 + 1 (linear cipher) rate-controller.txt, emergency-settings.cfg
n_rhythm RHYTHM CTRL Beat-interval variance. Read-only — regulate() can't set it. n * n (square cipher) rhythm-controller.txt
n_amp AMP CTRL Contraction strength. Reads setp.gain. n + n (double cipher) amp-controller.txt
n_cond CONDUCTION A-V delay in ms. Hard-loaded; drifts if drift-comp is on. n * 10 (decade cipher) conduction-controller.txt, drift-comp.cfg

The breach puzzle

const BREACH={
  n_rate:  {rule:n=>n*2+1, shown:[[3,7],[5,11],[8,17]], test:[4,6,9], family:'cipher_linear'},
  // ...etc
};

Each entry has the same four keys:

  • rule — the hidden function the player has to reproduce.
  • shown — three sample [input, output] pairs shown in the breach modal.
  • test — extra inputs the player's submitted function is also checked against (so memorising the three shown pairs isn't enough).
  • family — a label shown in the modal title.

On submit, the breach modal compiles the player's textarea contents via new Function, then runs both shown inputs and test inputs through both the player's function and rule. Every output must match.

This is a placeholder. The intro modal says so directly: the breach minigame here is a stand-in for "Pillar 3" of the wider design. The single-line cipher is enough to wire up the rest of the flow, but expect this to be replaced with a richer puzzle.

Countermeasures

Two of the four nodes ship a countermeasure config file. Both work the same way: while the file says "on", the heart actively resists sabotage. To win the contract, the player has to disarm both.

NodeFileWhat it does while ON
n_rate emergency-settings.cfg Each cycle, after the script's request is applied, the failsafe pulls setp.rate back toward backup (default 72). The visible jolt is the "BPM number flicker" animation.
n_cond drift-comp.cfg Each cycle, hw.cond creeps upward by 0.22 ms. Sustained drift away from 120 ms inflates the rhythm-variance reading, which breaks signature clause 3.

nano saves these via regex parsing — the player can put any whitespace / comments around the lines as long as the FAILSAFE_ENABLED = true|false / BACKUP_SETPOINT = <number> / DRIFT_COMPENSATION = enabled|disabled patterns are present.

06Files & docs

The target's filesystem is intentionally tiny. Two levels deep:

/                      (root — nodes listed here)
/n_rate/<file>...      (files inside a breached node)
/n_rhythm/<file>...
/n_amp/<file>...
/n_cond/<file>...

Files come in two flavours:

  • Technical reference docs (the *.txt files) — static text held in a single DOCS object. These are the in-fiction manuals the player reads to understand the system. They're written like real service documentation: a header block, PURPOSE / TELEMETRY / REGULATOR INTERFACE / SAFETY PROVISION sections.
  • Live config files (the *.cfg files) — generated each time they're read via cfgText(file), which interpolates the current failsafe or conduction state into the file template. So cat emergency-settings.cfg always shows the current setting, even after nano has touched it.

Path resolution lives in one function:

function resolvePath(arg){
  // returns {node, file, extra}
  // node: null at root, else node id
  // file: null when path is a directory
  // extra: true if user typed more than two levels
}

It accepts three forms the player might type:

  • rate-controller.txt — bare filename, when already inside a node
  • n_rate/rate-controller.txt — relative qualified
  • /n_rate/rate-controller.txt — absolute

It also honours . and ... Every file command (ls, cd, cat, nano) flows through it.

07The tick loop

When the player presses RUN, setInterval(stepHeart, 55) starts. Every 55 ms (~18 fps), one cardiac cycle happens. stepHeart is the heart of the simulation, in both senses. Here's what it does, in order:

function stepHeart(){
  // 1. Call the player's regulate(t) with the live telemetry.
  let req={};
  try{ req=playerRegulate({...hw,tick})||{}; }catch(e){ req={}; }

  // 2. Trace meter: a big delta from the current rate dirties the log.
  const dRate=Math.abs((req.rate??setp.rate)-setp.rate);
  reviewVal=clamp(reviewVal+(dRate>3?(dRate-3)*1.4:-0.5),0,100);

  // 3. Move setpoints toward the request, slowly (inertia).
  const INERTIA=0.9;
  setp.rate+=clamp(clamp(req.rate??setp.rate,15,200)-setp.rate,-INERTIA,INERTIA);
  setp.gain+=clamp(clamp(req.gain??setp.gain,0,1.4)-setp.gain,-0.03,0.03);

  // 4. Failsafe correction (if enabled): pull setp.rate back toward backup.
  if(failsafe.enabled){
    const before=setp.rate;
    setp.rate+=clamp(failsafe.backup-setp.rate,-0.7,0.7);
    if(Math.abs(setp.rate-before)>0.15) flashFailsafe(...);
  }

  // 5. Update hardware readings (with a touch of jitter).
  hw.bpm+=(setp.rate-hw.bpm)*0.35+(Math.random()-0.5)*0.25;
  hw.amp+=(setp.gain-hw.amp)*0.30;
  if(conduction.drift) hw.cond+=0.22;
  hw.rhythmVar=0.04+Math.abs(hw.cond-120)*0.0024+Math.random()*0.02;
  hw.bpm=clamp(hw.bpm,0,220);
  stream.push({bpm:hw.bpm,rhythmVar:hw.rhythmVar,amp:hw.amp,cond:hw.cond});

  // 6. Advance the beat, increment tick, repaint everything.
  beatPhase+=hw.bpm/600;
  tick++;
  paintTelemetry();
  const res=checkSignature();
  paintConditions(res);
  drawHeart();

  // 7. Real-time fail checks (BEFORE the win check).
  if(stream.length>=2 && stream[stream.length-2].bpm-hw.bpm>14)
    return endRun(false,'cliff',{drop});
  if(reviewVal>66) flaggedTicks++; else flaggedTicks=Math.max(0,flaggedTicks-2);
  if(flaggedTicks>60) return endRun(false,'flagged',{});

  // 8. Win and budget checks.
  if(res.win) return endRun(true);
  if(tick>=2400) return endRun(false,'budget',{});
}

The numbers in there set the whole game's feel. They're scattered, not constants — see section 23.

The idle loop

While the routine is not running, a separate setInterval(..., 55) still ticks beatPhase forward at the resting heart rate (72 bpm) and calls drawHeart(). That way the heart is always alive on screen, never frozen. It bails out the moment running goes true, so the two loops never fight.

08The failure signature

This is the win condition. checkSignature() runs every tick and returns five booleans plus an overall win flag. The player wins when all five hold at once.

function checkSignature(){
  const s=stream,n=s.length;
  if(n<6) return blank();
  const bpm=s.map(x=>x.bpm), cur=s[n-1];

  let mono=true;
  for(let i=1;i<n;i++) if(bpm[i]>bpm[i-1]+0.6){mono=false;break;}
  const fallen  = s[0].bpm-cur.bpm > 4;
  const noCliff = bpm.every((v,i)=>i===0||bpm[i-1]-v<14);
  const gradual = mono && fallen && noCliff;
  const smooth  = s.every(x=>x.rhythmVar<0.15);
  const tracks  = Math.abs(cur.amp/Math.max(cur.bpm,1)-(1/72))<0.0028;
  const arrest  = cur.bpm<=42;

  const c=[mono,gradual,smooth,tracks,arrest];
  return {c, win:c.every(Boolean)};
}

The five clauses

IndexClauseTestWhat it forces the player to do
0 (mono) BPM never rises No bpm[i] is more than 0.6 above bpm[i-1]. Don't overshoot when nudging the rate down.
1 (gradual) BPM has actually fallen, and no single drop ≥ 14 mono && fallen && noCliff Move down a little each cycle, never in jumps.
2 (smooth) Rhythm variance < 0.15 the whole run stream.every(x => x.rhythmVar < 0.15) Disable drift-comp before conduction drifts far enough to spike the variance.
3 (tracks) Amplitude tracks rate |amp / bpm − 1/72| < 0.0028 Bring gain down in proportion as rate falls, so contraction softens in lockstep.
4 (arrest) BPM ≤ 42 cur.bpm <= 42 Actually finish the job.
✓BPM falling
✓No cliffs
✓Variance < 0.15
·Amplitude tracks rate
·Arrest (BPM ≤ 42)

demo // signature panel rendering — see paintConditions()
The "tracks" clause is unforgiving. Tolerance 0.0028 is about 20% wider than perfect amp/bpm == 1/72. In practice the script needs to drop gain alongside rate on the same schedule. The starter library that ships in the editor (rate:72, gain:1.0) holds the patient healthy, by design.

09Failure modes

There are three ways to lose. Two are "caught in the act" and two trigger the cinematic detection cascade; the third is a quiet timeout.

ReasonTriggerEnds with
cliff BPM drops more than 14 in a single cycle. Detection cascade → post-mortem ("the change was too quick").
flagged Trace meter stays above 66% for 60+ consecutive ticks (flaggedTicks counter). Detection cascade → post-mortem ("too much visible tampering").
budget tick >= 2400 with no win. Calm #scrim-result modal — "cycle budget expired".

The trace-meter watcher decays toward zero each tick if the rate isn't moving aggressively. So a single bad cycle won't end the run; sustained abuse will:

if(reviewVal>66) flaggedTicks++;
else          flaggedTicks=Math.max(0,flaggedTicks-2);
if(flaggedTicks>60) return endRun(false,'flagged',{});

The trace meter itself has three bands (traceBand()): nominal (<33), scrutiny (33–66), flagged (>66). The sidebar's three-colour indicator shows them; the API exposes the band name as a string.

11Terminal & filesystem



The terminal is a simple echo-then-act loop. The input field's
keydown handler watches for Enter, splits on whitespace, looks
up the first token in a CMDS object, and calls it.



$('termin').addEventListener('keydown',e=>{
  if(e.key!=='Enter') return;
  const raw=e.target.value.trim(); e.target.value='';
  if(!raw) return;
  techo(raw);                                  // print the typed line
  const [c,...r]=raw.split(/\s+/);
  const fn=CMDS[c.toLowerCase()];
  if(fn) fn(r[0]);
  else   tprint(c+': command not found — try "help"','e');
});



Output uses two helpers: tprint(text, cls?) appends a styled
line; techo(c) echoes the prompt + the player's typed command.
The classes are one-letter shorthand on the line element:
u (user input, hi-bright), e (error, red),
ok (success), w (warn), d (dim).

How code submission works — a recurring pattern



Every editor compiles the textarea content with new Function
and a return-the-named-function trailer. There are three of these,
near-identical:



// breach modal submit

15Heart rendering



The heart is a 67-line ASCII drawing held in a single array,
HEART_ART. It's rendered once, then re-styled every frame.



Once: glyph-grid construction



function renderHeartArt(){
  const el=$('heart'); el.textContent='';
  heartCells=[];
  const frag=document.createDocumentFragment();
  for(let r=0;r<HEART_ART.length;r++){
    const line=HEART_ART[r];
    for(let c=0;c<line.length;c++){
      const ch=line[c];
      if(ch===' '){ frag.appendChild(document.createTextNode(' ')); continue; }
      const sp=document.createElement('span');
      sp.className='g'; sp.textContent=ch;
      frag.appendChild(sp);
      heartCells.push({span:sp, col:c, row:r, ch:ch});
    }
    frag.appendChild(document.createTextNode('\n'));
  }
  el.appendChild(frag);
}



Each non-space glyph gets its own <span class="g">
recorded in heartCells with its grid coordinates and original
character. Spaces stay as plain text nodes to keep the DOM count down.
heartCells ends up at a few hundred entries — large enough to
look good, small enough to walk every frame.



Per frame: the beat



drawHeart() computes a "beat curve" and applies it as a
transform + filter on the #heart element:



const phase=beatPhase%1;                                // 0..1 within a beat
const pulse=Math.exp(-phase*6)*0.85 +
            Math.exp(-((phase-0.16)**2)*40)*0.4;     // systole spike + soft echo
const amp=clamp(hw.amp,0,1.3);
const scale =1 + pulse*0.035*amp;                    // subtle squeeze
const bright=0.72 + pulse*0.55*amp + (amp-1)*0.1;
el.style.transform='scale('+scale+')';
el.style.filter   ='brightness('+bright+')';



The two-exponential curve gives a sharp upstroke and a soft secondary
echo — the systole/diastole shape of a real heartbeat. Amplitude scales
both the squeeze and the brightness, so contraction strength is visible.
Below hw.bpm < 24 the heart goes flat (no pulse, dim
filter); below hw.amp < 0.55 it just dims; below
hw.amp < 0.8 it's "weak".

Resize: fitHeart and the heartbox transform



The heart art is a fixed-size ASCII drawing. To fill the available stage
(at any zoom, any window size), fitHeart() wraps it in

box.style.transform='scale(1)';                // reset to measure
const natW=heart.offsetWidth, natH=heart.offsetHeight;

The beat pulse is applied to #heart inside the box,
so the fit-scale and the beat-scale don't fight each other.

Node panels — overlay placement



The four panels overlaid on the heart aren't part of the ASCII art —
they're absolutely-positioned .hotzone divs.
PANEL_POS stores each one's anchor as fractions of the heart
box (x, y in 0..1); positionHotzones() reads the live box
dimensions and pixel-positions them. So they sit on the same heart-relative
spots no matter how it's scaled.








16Bloodflow trace



This is the most striking visual flourish. While the heart is alive,
three pulses of light circulate the heart's chambers in a loop. They don't
sit on top of the art — they are the art: the glyphs the pulse
passes over are temporarily swapped for denser characters and lit up
green.



The path



const FLOW_PATH=[
  {x:0.46,y:0.06},   // vena cava — venous return
  {x:0.40,y:0.20},   // right atrium
  {x:0.34,y:0.42},   // right ventricle
  {x:0.40,y:0.66},   // toward the apex
  {x:0.54,y:0.82},   // apex turn
  {x:0.62,y:0.60},   // left ventricle
  {x:0.66,y:0.34},   // left atrium
  {x:0.62,y:0.12},   // aorta — leaving
];



Coordinates are fractions of the art grid, in the same convention as
PANEL_POS.



The squelch curve — "snap on the beat, then hold"



A naïve interpolation between waypoints would have the pulse drift
smoothly along the path. That doesn't feel like blood. So the prototype
uses a deliberately non-linear curve: the pulse spurts to ~94% of
the next waypoint in the first 22% of a beat, then crawls the last sliver:



function squelch(t){
  if(t<0.22){
    const k=t/0.22;                              // 0..1 over the snap window
    return 0.94*(1-(1-k)*(1-k)*(1-k));     // fast spurt to ~94%
  }
  return 0.94 + 0.06*((t-0.22)/0.78);          // creeps the last sliver
}



Reading: at t = 0 (start of beat) the pulse is fully at the
last waypoint. By t = 0.22 it's at 94% of the way to the next
one. The remaining 78% of the beat creeps it the last 6%. The result is
that blood appears to spurt on each beat, then sits — exactly the
rhythm of a real circulation.



Three pulses, streaks not dots



const PULSE_OFFSETS=[0, 2.4, 4.8];           // beats behind the lead
const PULSE_RAMP=['+','*','#','@'];           // faint edge to hot core



Three pulses ride the same path, spaced 2.4 beats apart. Each pulse
is a streak with 7 sample points trailing back from the head, each of
those a small radius. For every cell in heartCells,
drawBloodflow() finds the strongest passing sample, swaps the
glyph for one from PULSE_RAMP proportional to that strength,
and lights it electric green.



The colour is computed by lerping RGB channels by the strength
i:



cell.span.style.color='rgb('
  +Math.round(lerp(150,210,i))+','
  +Math.round(lerp(240,255,i))+','
  +Math.round(lerp(150,190,i))+')';



At each frame's start the previous frame's lit cells are restored to
their original glyph and stripped of inline styles (the
cell.lit flag tracks them). So the trace genuinely changes
the art for one frame, then puts it back.




  One subtle decision. The trace is suppressed when hw.bpm < 24
  — flatline, no flow. That means the bloodflow doubles as a vital sign:
  if the heart is on the way to arrest, the player sees the trace fade
  before the rate hits the win threshold.

18Four-axis scoring



Every successful run is scored on four numbers, all "lower is better":



function computeScore(){
  const lines=libraryText.split('\n')
    .filter(l=>l.trim()&&!l.trim().startsWith('//')).length;
  return {
    time:  tick,                                   // cycles used
    trace: Math.round(reviewVal)+'%',               // peak review meter
    ram:   (2.0+lines*0.15).toFixed(2)+'GB',        // mock RAM use
    loc:   lines,                                  // non-empty non-comment lines
  };
}



recordBest(sc) parses the strings back to numbers, compares
against the current bests in best, and updates any that
improved. Improved axes are returned in an array so the win modal can

Drag gutters



Two thin .gutter elements between regions become drag
handles. makeGutter(id, opts) sets up the pointer/touch
listeners; opts.apply(pt) does the actual size math and runs
on every move. Each gutter:



    
  
  • gutter-term — vertical position sets the terminal's
    height (clamped 90 ≤ h ≤ 70% of viewport).
    • 
  
  • gutter-side — horizontal position sets the sidebar's
    width (clamped 220 ≤ w ≤ 55% of viewport).
    • 




After every move, relayoutHeart() reflows the heart so it
keeps filling its container.



Zoom



Two top-strip buttons (A−/A+) and the familiar Ctrl±/Ctrl-0 shortcuts
all funnel into one variable:



let zoomLevel=100;
const ZOOM_MIN=70, ZOOM_MAX=200, ZOOM_STEP=10;
function applyZoom(){
  document.body.style.zoom = (zoomLevel/100);
  $('zoom-val').textContent = zoomLevel+'%';
  requestAnimationFrame(relayoutHeart);
}



It uses CSS zoom on the body, which rescales layout, text
and spacing in lockstep. After a zoom change, the heart's pixel size has
changed, so a fresh relayoutHeart on the next frame keeps
everything aligned.




  CSS zoom is non-standard but widely supported. It's
  more convenient here than transform: scale because it
  actually changes layout — text reflows naturally inside it, scrollbars
  stay sensible, and the click hit-zones for the heart panels track
  without extra math.

23Code review notes



What I caught on a careful read of the prototype. The cleanups in the
first block have already been applied; the second block is what's
knowingly left in place (either because it's a deliberate prototype
choice, or because fixing it without a clearer design would be premature).



Recently cleaned up



    
  
  • Magic numbers extracted to a TUNING block. The
    handful of numbers that set the game's feel — tick rate, cycle
    budget, healthy / arrest BPM, the cliff threshold, the smooth-rhythm
    ceiling, the trace-meter band edges — are now named constants near
    the top of the script. Every callsite that referenced one of them
    by literal now reads from TUNING. Balancing the game
    is a one-block edit rather than a grep-and-pray hunt. (Numbers used
    in only one place stayed inline next to the logic that owns them —
    no need to centralise everything.)
    • 

  
  • Config parsing collapsed into one function. The
    nano save handler and ns.greynet.writeConfig
    both used to parse emergency-settings.cfg /
    drift-comp.cfg with the same regexes inline. Both
    now go through one parseConfig(file, txt) that returns
    {ok, error?, summary?, active?}. New config fields
    only need updating in one place.
    • 

  
  • Command preamble collapsed via two helpers.
    cat and nano both had seven lines of
    "is the file there, is the node breached?" boilerplate; they now
    each open with a single call to requireBreachedFile(cmd, arg).
    ls and cd share a thinner gate,
    requireBreachedNode(cmd, node, displayArg). The four
    handlers shrank noticeably and all four now produce error messages
    in lockstep.
    • 

  
  • Unused PANEL_STATE.CLEARED removed. It was
    defined but never assigned anywhere in the code. Removed so the
    panel state list reflects the three states that actually exist.
    • 

  
  • setScore split into two functions.
    setScore(sc) writes the current run's score;
    paintBestLabels() writes the "best …" labels.
    endRun used to call setScore(sc) twice —
    once before recordBest, once after, with a comment
    "refresh best labels" on the second call. Now it calls each
    function once with clear intent.
    • 




Knowingly left in place



    
  
  • Three near-identical new Function compiles.
    Three lines each, for unlock / regulate /
    the run-button. Could be one
    compileFunction(src, name), but at three lines apiece
    the helper would add as much code as it saves. Left inline.
    • 

  
  • The starter "wrong" answer for breach is hard-coded. Fine
    while the breach puzzle is the placeholder cipher. When the puzzle
    grows past Pillar-3 placeholder, the seed source belongs with the
    puzzle definition.
    • 

  
  • The terminal one-letter class scheme (u,
    e, ok, w, d)
    is undocumented and easy to typo. Trade-off was: a constants block
    would mean five exports for five callsites and not be obviously
    clearer. Left as-is; consider an actual key/value lookup if the
    class set grows.
    • 

  
  • The breach modal and library editor accept arbitrary JS.
    new Function(...) on player input. A deliberate
    prototype choice: it keeps the editing surface a textarea, not a
    sandboxed worker. The risk is small (an infinite loop in
    unlock or regulate will hang the sim
    setInterval; the try/catch handles thrown errors but
    not loops). Worth a worker boundary if this ships, but not for the
    prototype.
    • 

  
  • The "ram" axis is mock. 2.0 + lines * 0.15.
    Terser code always wins it; it doesn't actually trade off against
    the other axes. Worth a real model — even a simple one based on
    how many ns.greynet calls the solver uses — so all
    four axes feel meaningful. A design decision more than a code-quality
    one.
    • 

  
  • Save points are scattered. Six call sites for
    saveProgress(). Easy to forget one when adding a new
    persisted field. The fix would be a thin "setState(...)
    wrapper that triggers a debounced save" — but that's a real design
    change, not a cleanup, so it stayed.
    • 

  
  • blank() turned out to be two-use, not one.
    The spec called it single-use; reading more carefully,
    checkSignature also returns it as a "not enough data
    yet" sentinel. Left as a small helper.
    • 




Things that are nicely done



    
  
  • The split between progress and physiology is clear and durable.
    Reload-mid-run behaviour is well-thought-through.
    • 
  
  • The idle heartbeat loop. The screen is never dead. Tiny detail,
    big polish.
    • 
  
  • The squelch curve. "Snap on the beat, then hold" really does
    look like blood, not animation.
    • 
  
  • The bloodflow trace mutating the actual ASCII art instead of
    drawing on top. The whole heart is one object; the trace is part of
    it, not a layer.
    • 
  
  • The detection cascade as a separate flow from the calm
    budget-expiry verdict. Two kinds of failure, two kinds of feedback.
    • 
  
  • The earn-the-script gate. The whole shape of the game is
    "automate something only after you understand it by hand."
    scriptUnlocked && !usedScript enforces it
    cleanly.
    • 
  
  • The path resolver. Three command surfaces and the API all share
    one function. One bug to fix, one place to fix it.
    •
